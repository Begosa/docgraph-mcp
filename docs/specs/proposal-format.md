# Proposal Mutation Format

A proposal contains ordered mutations. It is validated and committed atomically.

## Canonical ops

```text
upsert_node
add_alias
add_edge
upsert_claim
attach_evidence
mark_claim_status
```

Use `dg_mutation_schema` when unsure.

## Shared-knowledge metadata

`upsert_node`, `add_edge`, and `upsert_claim` may include:

```json
{
  "visibility": "shared",
  "finder_role": "rtl",
  "audience_roles": ["rtl", "firmware", "test_debug", "architecture"],
  "interface_tags": ["data_path", "channel_count"]
}
```

Allowed visibility:

```text
local | shared | global | shared_candidate
```

Use this metadata to prevent useful findings from staying trapped in the finder role.

## Accepted aliases

The MCP accepts common aliases to reduce agent brittleness, but curator prompts should prefer canonical names.

```text
node / add_node / create_node     -> upsert_node
alias / upsert_alias              -> add_alias
edge / upsert_edge                -> add_edge
claim / add_claim                 -> upsert_claim
evidence / claim_evidence         -> attach_evidence
status / claim_status             -> mark_claim_status
```

## Evidence-first rule

Active non-open-question claims require active supporting evidence after the proposal is applied.

Canonical explicit order:

```text
upsert nodes/aliases/edges
upsert claims
attach evidence to claims
```

If supporting chunks do not exist yet, the curator must first ingest the source or investigation report.
Use the exact `chunk_ids`/`chunk_refs` returned by ingestion. Chunk IDs are random backend IDs, not a pattern derived from source IDs, episode IDs, paths, or chunk numbers.

## Evidence shortcut

Compact claim form is supported:

```json
{
  "op": "upsert_claim",
  "claim_id": "claim.process_x.channels",
  "target_node_id": "block.process_x",
  "claim_text": "PROCESS_X has 8 input channels and 16 output channels.",
  "classification": "Fact",
  "confidence": "medium",
  "visibility": "shared",
  "finder_role": "rtl",
  "audience_roles": ["rtl", "firmware", "test_debug", "architecture"],
  "interface_tags": ["data_path", "channel_count"],
  "chunk_ids": ["chunk_id_from_ingest_chunk_refs"]
}
```

The backend expands `chunk_ids` into `attach_evidence` mutations before validation/commit.
Proposal references are preflighted, so a missing target node, target edge, claim, or chunk ID fails with a typed validation error before SQLite foreign-key enforcement.

## Generated docs exclusion

Do not ingest generated graph output as evidence:

```text
docs/rendered/*
generated architecture.md
files marked Generated by render_docs
```

Rendered docs are outputs of the graph, not source evidence.
